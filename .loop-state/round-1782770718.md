# Hourly E2E Loop · 2026-06-29 22:05 UTC

## Story picked
**"Prepare an application package"** — product-spec.md § 5 "客户端半自动投递", routed through the
`prepare_application` LangGraph saga (delivery-loop-plan.md § 3, north-star metric TTAR).

A user POSTs `/applications/prepare` with `{jd_url, base_resume_id, base_resume_content, form_fields}`.
The Coordinator drives a fixed StateGraph:

```
parse_jd  →  customize_resume  →  cover_letter  →  form_answers  →  finalize
                       (fabrication_guard wraps customize)
```

Each stage has a saga branch: parse_jd fail short-circuits to finalize; customize/cover/form
fall back instead of failing. TTAR is recorded in `agents/harness/ttar.py`.

## Live story probes

### Probe 1 — happy-ish prepare (live Greenhouse JD)
```
POST /applications/prepare {jd_url: greenhouse/synthetic/4071234, …}
=> 200 parse_jd=ok customize_resume=failed (Redis 6380 ECONNREFUSED)
   cover_letter=fallback form_answers=fallback
```
**FINDING**: `agents/tools/auto.py::redis_get` / `redis_setex` do NOT degrade gracefully on
`ConnectionRefusedError`. They open a connection unconditionally and let `redis.ConnectionError`
propagate. The customize node then catches it as a saga stage failure and falls back to base —
but the workflow could have run perfectly fine with no cache (the cache is a Redis-backed
optimization, not a correctness requirement). This bug single-handedly drops customize success
on every dev box without Redis.

### Probe 2 — trace continuity
```
curl -i -H "X-Trace-Id: 11111111-2222-3333-4444-555555555555" http://127.0.0.1:8000/healthz
=> x-trace-id: 11111111-2222-3333-4444-555555555555    ✅
curl -i -H "X-Trace-Id: 11111111-2222-3333-4444-555555555555" http://127.0.0.1:3001/api/health
=> x-trace-id: 11111111-2222-3333-4444-555555555555    ✅
```
Inbound UUID-formatted trace propagates verbatim through both layers. Garbage payloads
("test-trace-001") get replaced with a fresh UUID — correct (clients can't inject log noise).

### Probe 3 — error envelope
```
POST /applications/prepare {no body, no auth}
=> 401 {error: {code: AUTH_REQUIRED, traceId, traceCode: R-NPA0, messageKey,
                                trace_id (legacy alias)}}
```
**FINDING**: Envelope conforms to schema EXCEPT `action` is missing for all `HTTPException`
paths. Per `docs/architecture/error-handling.md` § 1 P4, every code must populate `action` so
the UI can render a CTA. AUTH_REQUIRED → `{kind: "reauth", redirect: "/auth"}`. Currently
only `BudgetExhausted` (LLM_BUDGET_EXHAUSTED) and the catch-all `INTERNAL` get an action.

### Probe 4 — idempotency
```
POST /applications/prepare {application_id: 44…44} twice
=> both 200, same application_id (44…44)
```
Idempotency on application_id holds — the workflow uses the caller-supplied id and never
double-creates a draft row. ✅

### Probe 5 — locale handling
```
POST … -H "X-Relay-Locale: zh"
=> response has no X-Relay-Locale header echo back
```
**FINDING**: `prepare-application` endpoint doesn't even read X-Relay-Locale. The cover letter
will always be English, regardless of the UI locale. The endpoint should at minimum echo back
the inbound locale so the web layer renders the response in the same tongue.

## Why not 100

- Cost-guard live probe (-2 pts): the story-level cost regression test would need a stub LLM that emits a giant
  fake usage_metadata to trip `BudgetExhausted` from within `prepare_application`. Existing tests cover
  `post_model_hook` raising `BudgetExhausted` (test_cost_tracker.py, test_context_compaction.py) — the marginal
  gain from a story-level probe is small and the saga already proves the workflow handles the raised
  `BudgetExhausted` via its `try/except` in `_customize_resume_node`. Deferring to round 21+.
