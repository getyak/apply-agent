# agent-diff-guard 本地控制台深度产品与体验测评

> 测试地址：`http://localhost:4757/`  
> 测试日期：2026-06-21  
> 测试对象：`agent-diff-guard · 控制台`  
> 说明：页面实际源码位于本机相邻仓库 `personal/agent-diff-guard`，本报告存放在当前工作区，仅做评审与需求输入，未修改目标应用。

## 1. 执行摘要

### 1.1 一句话结论

产品的核心命题成立：当一个开发者并行使用多个 coding agent 时，真正稀缺的是“只把必须由人判断的 1–3 个风险送到眼前”，而不是更多扫描结果或更多仪表盘。

当前版本已经具备可信的技术骨架，但控制台的功能边界明显扩张过快。审查队列、守门记录、危险地图是核心价值；成本、项目、Loop、晨报、AI 问答各自有潜力，但目前共同挤在一级导航中，削弱了“关键时刻刹车”的聚焦感。产品文档明确写着“不做 dashboard”，当前页面实际上已经接近一个综合型 agent dashboard。

### 1.2 当前成熟度判断

| 维度 | 评分 | 判断 |
|---|---:|---|
| 核心问题价值 | 9/10 | 多 agent 场景下确实存在注意力与审计缺口 |
| 核心工作流完整度 | 7/10 | 发现 → 解释 → 人工裁决 → 终端信箱链路已成形 |
| 信息可信度 | 5/10 | 演示/真实数据混杂，隐私口径有冲突，成本数据疑似失真 |
| 可用性 | 6/10 | 桌面端可用，导航清楚；高频决策仍需减少认知负担 |
| 移动端 | 2/10 | 390px 下固定侧栏挤压内容，页面基本不可用 |
| 产品聚焦 | 5/10 | 一级模块过多，核心价值被成本、项目、Loop 等功能稀释 |
| 工程质量 | 8/10 | 414 项自动化测试全部通过，API 错误语义整体良好 |
| 商业化准备度 | 4/10 | 单机 dogfood 可用；团队身份、策略、审批与证据链仍不足 |

### 1.3 最优先处理的五件事

1. 修复数据可信度：成本数量级、晨报预算语义、真实/演示数据隔离。
2. 修复隐私承诺冲突：任务原文在越界记录、队列和信箱中可见，但局部页面又声称“原文已 hash、不留存”。
3. 把首页和一级导航收缩到“待处理风险”主线，其他能力降到二级或高级模式。
4. 增加仓库/分支/提交/agent 身份和“当前是否仍可处置”的明确上下文。
5. 移动端至少提供可阅读、可紧急审批的响应式布局。

### 2.2 未执行的动作

- 没有向 DeepSeek 发送问题。打开 Ask Guard 后确认了联网披露，但未触发外部数据传输。

- 没有启动 daemon 消费信箱，因此“终端取走 → 执行 → 回写结果”的真实闭环仍需单独端到端验收。

## 3. 产品定位：哪些功能真的有必要

### 3.1 必须做深的核心功能

#### A. 审查队列

这是产品存在的理由，也是最值得持续投入的界面。用户真正要回答的是：

1. 为什么此刻打断我？
2. 这项改动是否仍然存在、是否仍可处置？
3. 风险证据是什么？
4. 放行、驳回或误报之后会发生什么？
5. 谁做了决定，决定是否可追溯？

目前已经有风险级别、文件、规则、任务偏离度、diff 片段和三种裁决，方向正确。下一步不应该继续增加信息块，而应提升证据可信度和裁决效率。

#### B. 守门记录

这是团队版和企业版的基础资产。它不只是历史列表，而应成为“agent 改动—规则命中—人类决定—最终执行结果”的证据链。

当前表格支持筛选和行内展开，实用；但缺少 actor、commit SHA、agent/session、最终结果、审批理由和可导出证据包。

#### C. 危险地图

这是最具差异化的功能之一：把事后发现转成 agent 编码前的约束。比单纯的统计看板更有复用价值。

建议保留，但默认展示应只包含当前仓库的真实数据；演示仓库必须显著隔离。复制内容还应显示版本、生成时间和适用范围，防止旧规则被长期复用。

### 3.2 有价值，但应降级为二级功能

#### D. 规则洞察

“哪条规则误报最多”有价值；“规则”作为页面名称却暗示可以管理规则，实际上当前只能查看统计，添加规则仍需编辑源码。

短期建议改名为“规则效果”或“规则洞察”。只有支持仓库级启停、级别调整、例外、测试样例和版本记录后，才应叫“规则管理”。

#### E. 越界记录

确定性 policy 违规是强信号，适合复盘与合规。但“只记录、不阻断”不应是固定产品行为；对于冻结窗口、生产密钥、受保护目录，团队通常需要可配置的阻断策略。

建议作为“策略与违规”子模块，并明确区分：守门风险、policy 违规、执行器安全拦截。这三类信号目前在用户心智中容易混淆。

#### F. 终端信箱与执行记录

闭环价值高，但当前同时出现“终端信箱”和“执行记录”，且 5 条指令长期待执行、0 条已执行，会让用户怀疑系统没有真正闭环。

建议合并为“处置任务”，用状态机表达：待领取 → 已领取 → 执行中 → 成功/失败/被安全闸拦截 → 已复核。信箱是实现细节，不应成为用户必须理解的产品概念。

### 3.3 暂时不应占据核心导航的功能

#### G. 用量与成本

这是另一个产品方向，不是守门主线。它回答“烧了多少钱”，而守门回答“哪一处必须由人看”。两者用户意图、数据来源和购买理由都不同。

除非能建立“风险减少/事故避免/人类审查时间”与成本的直接关系，否则建议移到“分析”二级入口，甚至拆成独立产品。当前 $11,179、单日 14 亿 token 等数据会首先引发对统计正确性的质疑。

#### H. 项目管理

项目注册是 daemon 路由所需配置，不是核心日常任务。应放到设置中。`bypassPermissions · 完全自主` 是高风险选项，当前呈现得像普通下拉项，缺少影响说明、二次确认和最小权限引导。

#### I. Loop 监控与晨报

它们面向“长时间无人值守的自治 loop”，是高级场景，不应与基础 pre-push 守门平级。当前只有一个 session，页面信号密度不高，且预算 0% 的含义不清。

建议做成可选插件/高级模式。只有用户创建或接入 Loop Contract 后才显示相关导航。

#### J. Ask Guard

AI 问答适合作为解释层，不应成为决策真相源。它真正有用的三个固定任务是：

- 用自然语言解释为什么拦截。
- 合并同一根因的多个 finding。
- 基于本地证据生成可编辑的处置建议。

不建议扩展为通用聊天。固定问题和结构化回答比空白输入框更可靠、更便宜、更容易审计。

## 4. 用户视角：我真正想看到什么

### 4.1 首页只需要回答四个问题

1. 现在有没有必须由我处理的风险？
2. 最严重的是哪一项，为什么？
3. 哪些 agent/repo 正在被保护，守门是否在线？
4. 我做完处理后，push/merge/loop 是否真的恢复？

目前首页同时展示扫描总数、wake 命中、自动放行率、仓库数、14 天趋势、活动流、提醒和悬浮 AI。信息不算错误，但“今日态势”仍偏运营看板，而不是决策收件箱。

建议首页结构改为：

- 第一屏：`2 项需要你决定`，直接显示最严重两项和预计处理时间。
- 第二屏：守门健康状态，包括 hook/MCP/daemon 最后心跳与数据新鲜度。
- 第三屏：最近 7 天质量信号，仅保留误报率、重复风险、平均处理时长。
- 其余趋势和成本进入分析页。

### 4.2 每个 finding 应补齐的上下文

- 真实/演示/历史/实时来源标识。
- 仓库、分支、commit SHA、工作树状态。
- 产生改动的 agent、session、任务来源。
- “为什么此刻打断”的确定性证据和置信度。
- finding 是否仍存在，最后验证时间。
- 处置动作的具体副作用。
- 规则历史准确率与该仓库例外。
- 可直接打开本地 diff、文件和对应 session 的入口。

### 4.3 裁决体验应该是什么样

`放行` 已经要求填写理由并再次确认，这是正确设计。但三个动作应进一步语义化：

- 放行本次：只针对当前 commit/finding，不永久修改规则。
- 添加例外：明确作用域和有效期，例如本仓库、本路径、7 天。
- 驳回并生成修复任务：先预览命令/提示词，再选择目标项目与执行权限。
- 标记误报：要求选择原因，例如路径规则太宽、任务本来就需要、证据失真。

不要把“标记误报”当成简单的第四个状态。它是规则学习的高价值标注，应被结构化采集。

## 5. 分页面功能测试结果

### 5.1 总览

**有效：** 核心 CTA 明确；风险数量与队列一致；最近活动可快速建立态势；定位句比旧版清楚。

**问题：**

- `34 总扫描 / 26 wake / 41.2% 自动放行率` 显示系统频繁打断，与“宁可漏、不可烦”的承诺存在张力。
- “本机 events.jsonl”属于实现细节，对普通用户没有决策价值。
- 右侧提醒和悬浮 Ask Guard 同时出现，遮挡活动内容并争夺注意力。
- 缺少“数据更新时间”和各采集器健康状态。

**结论：** 可用，但需要从指标看板收缩为决策首页。

### 5.2 审查队列

**有效：** WAKE/LOOK/已处理筛选存在；列表与详情并排；显示任务偏离度、理由、diff；放行有理由输入和确认步骤。

**问题：**

- 三条当前内容均标为“演示”，但页面没有总级别的 DEMO 模式提示，容易被误当成真实待处理风险。
- 筛选按钮缺少明显的可访问选中状态，DOM 中也没有呈现 `aria-pressed` 语义。
- 项目选择放在裁决区，但没有说明它只影响发回终端，还是也影响放行留痕。
- 仅凭百分比展示“任务 ↔ 改动偏离度”，没有解释模型、阈值和不确定性。
- 未显示当前 finding 是否仍可在工作树中复现。

**结论：** 这是最值得打磨的核心页面，功能方向正确。

### 5.3 守门记录

**有效：** 表格密度合理；状态筛选实用；点击行可展开仓库、范围、文件数和命中详情。

**问题：**

- 历史记录没有 actor、agent、session、commit SHA 和审批理由。
- 仅用 `HEAD~1..HEAD` 等相对范围，时间一久后无法稳定复现。
- “人工放行”筛选有入口，但样本中没有展示完整的人类决策证据。
- 长表格缺少日期/仓库/规则搜索和分页或虚拟化策略。

**结论：** 是团队/合规版核心资产，需要从“扫描流水”升级为“不可抵赖证据链”。

### 5.4 规则

**有效：** 能看到规则级别、命中、wake 和误报数；明确解释误报反馈的价值。

**问题：**

- 页面叫“规则”，但不能创建、编辑、停用、测试或限定作用域。
- “命中占比”列在 DOM 中没有可读数值，主要依赖视觉条形，信息不完整。
- 误报为 0 不等于规则准确，可能只是用户从未标注。
- 添加规则仍需改 `src/rules.ts`，对非作者用户不可用。

**结论：** 当前应改名“规则效果”；真正的规则管理应后续独立设计。

### 5.5 越界记录

**有效：** 明确区分确定性检测与 AI；能看到 policy、文件、时间、分支和任务。

**问题：**

- 页面公开展示任务原文，而守门记录详情又声称任务原文已 hash、不留存，隐私心智冲突。
- “零 AI”不等于“零误判”，路径匹配仍可能有规则误配。
- 当前固定“只记录、不阻断”，无法覆盖发布冻结等必须阻断的策略。
- 展示了 `.env` 等敏感路径，虽无正文，也应提供一键隐藏敏感元数据的演示/分享模式。

**结论：** 功能有必要，但需要统一隐私模型和策略执行级别。

### 5.6 危险地图

**有效：** “事后风险 → 事前 agent 上下文”的产品逻辑强；复制与 MCP 两种投喂方式合理；数据不含代码正文。

**问题：**

- 当前同时出现真实仓库和 `acme-corp/backend-api` 演示仓库，缺少明显隔离。
- “已连接”没有最后调用时间、客户端数量、版本或失败状态。
- 热度没有展示计算值、数据窗口和置信度，只给高/中/低。
- 复制给 agent 后没有反馈该上下文是否被实际使用。

**结论：** 应保留并做深，是比通用 dashboard 更有差异化的方向。

### 5.7 用量与成本

**有效：** 今日、项目、历史、session 四层下钻结构完整；明确标注“估算，非账单”。

**问题：**

- 单日 14 亿 token、项目累计上万美元的数量级极不寻常，应优先确认缓存 token 是否重复累计、消息快照是否累加、模型单价是否正确。
- `今日 1.8M token / 12m / $9.9` 缺少 input/output/cache 的可审计明细。
- 项目名存在路径压缩/归一化，用户难以对应真实仓库。
- 页面与核心守门任务关系弱，且 `/api/daily/list` 本次耗时 2.41s。

**结论：** 暂时降级为实验性分析功能。数据可信前不要用于预算决策。

### 5.8 项目

**有效：** 支持路径浏览、收藏、描述与权限模式；注册后可供 daemon 路由。

**问题：**

- `default · 只读观察`、`acceptEdits · 自动编辑`、`bypassPermissions · 完全自主` 的风险差异解释不足。
- 高危权限无需二次确认，也没有按项目展示 daemon 最后心跳。
- 一个示例项目使用不存在于本机语境的 Linux 路径，真实/演示混杂。
- 项目配置应位于设置，不应占一级导航。

**结论：** 必需的系统配置，但不是高频产品功能。

### 5.9 终端信箱

**有效：** 把面板裁决落盘并由 MCP 拉取，设计上避免复制粘贴；指令可审计。

**问题：**

- 5 条指令长期“待终端取走”，没有说明 daemon 离线、MCP 未连接或消费失败。
- 指令正文包含原任务片段和本地路径，和“只记 hash”的承诺冲突。
- 用户不能取消、重试、改派、设置过期时间或查看领取者。
- 自动生成的 `git checkout origin/main -- file` 属于破坏性恢复命令，虽然执行器会拦截，但 UI 不应制造必然失败的闭环。

**结论：** 闭环价值高，但当前状态机和隐私表达不完整。

### 5.10 执行记录

**有效：** 设计了成功、失败、拦截、待执行筛选，以及耗时和输出详情承载位。

**问题：**

- 当前 `0 已执行 / 5 待执行`，无法证明闭环真的工作。
- 信箱和执行记录重复展示同一批任务，信息架构冗余。
- 缺少 daemon 健康状态、领取时间、执行主机、权限模式、重试次数。

**结论：** 与终端信箱合并为单一“处置任务”页面。

### 5.11 Loop 监控与晨报

**有效：** session、漂移、预算、裁决和回滚点组成了自治 loop 所需的关键信号；晨报目标清晰。

**问题：**

- 页面显示预算 `0%`，但无法判断是“未配置预算”还是“已用 0%”；二者风险完全不同。
- 晨报在缺少目标关键词、预算语义不明时仍建议 `continue`，结论过于乐观。
- 回滚点显示 `HEAD`，不是稳定、可复制、可审计的 commit SHA。
- 仅 1 次迭代时给出趋势和正式建议，证据不足。
- Loop 就绪 L2 的计算为等权能力计数，容易制造“看起来已就绪”的错觉。

**结论：** 高潜力高级功能，但必须先建立保守的建议规则和稳定证据。

### 5.12 全局搜索与 Ask Guard

**有效：** `⌘K` 入口、快捷跳转、finding 搜索和自然语言提问整合合理；搜索 `secret` 能返回对应 finding；Ask Guard 明确披露 DeepSeek、元数据和“含代码”开关。

**问题：**

- 快捷跳转缺少项目、Loop 监控和晨报，和一级导航不一致。
- 搜索结果仍会暴露任务原文摘要。
- AI 披露写死 `api.deepseek.com`，若后端允许自定义 base URL，则 UI 可能显示错误供应商。
- “DeepSeek 已连接”只代表配置存在，不代表最近调用成功。

**结论：** 搜索值得保留；AI 应保持结构化、可追溯、只提案。

## 6. 横向问题

### 6.1 P0：隐私承诺互相冲突

目前至少存在三种口径：

- 守门记录声称任务原文已 hash、不留存。
- 审查队列和越界记录展示任务原文或摘要。
- 终端信箱把任务片段写入本地持久化指令。

这不一定代表实现泄漏，也可能是数据源不同，但产品没有解释数据生命周期。对于以“代码不离开本机、审计可信”为卖点的工具，这是阻塞信任的 P0。

需要定义统一的数据分类表：字段、来源、是否持久化、保存位置、保留期限、是否上云、脱敏策略、删除方式。UI 中的所有隐私文案必须从同一份能力状态生成，不能手写多套承诺。

### 6.2 P0：数据可信度不足

- 成本数据疑似数量级错误。
- “演示”与真实数据混在同一队列、危险地图和项目列表。
- 晨报建议没有把预算缺失、证据不足作为不确定性。
- `HEAD` 被当成回滚点，无法稳定复现。

安全产品一旦数字不可信，用户会连正确的 finding 一起忽略。应给每个数字显示：数据源、时间窗口、更新时间、估算/实测、采集完整度。

### 6.3 P0：处置闭环没有被证明

当前 5 条 pending、0 条 executed。产品声称“面板裁决 → 指令 → 终端 agent 执行”，但用户看到的是积压。

应在全局顶部显示：daemon 在线/离线、最后心跳、当前项目、待领取数量、最老任务年龄。离线时禁止使用“发回终端修复”这种确定语气，改为“保存为待办，daemon 上线后执行”。

### 6.4 P1：移动端不可用

390 × 844 实测中：

- 约 208px 固定侧栏始终占据左侧。
- 内容区只剩狭窄竖条，标题和卡片逐字换行。
- 搜索框溢出，表格和关键信息无法正常阅读。
- 没有折叠菜单或移动端底部操作栏。

建议最小目标不是完整移动管理，而是支持“紧急查看 + 放行/驳回”。小于 768px 时隐藏侧栏，以抽屉替代；finding 详情改单列；裁决动作固定底部；表格改卡片或允许明确横向滚动。

### 6.5 P1：首页与产品原则冲突

设计文档强调“不做 dashboard”，但当前共有 11 个一级页面和多组趋势/KPI。建议重新设定一级信息架构：

- 待处理
- 记录
- 策略
- 分析
- 设置

危险地图放策略，违规放记录，成本放分析，项目放设置，Loop 仅在启用后出现。

### 6.6 P1：首屏性能

`/api/findings` 本次耗时 4.32s，`/api/daily/list` 2.41s。首次浏览器导航曾超过 10 秒才稳定完成。

首页不应等待所有模块。建议：

- 首屏只请求 overview、当前 findings 和 health。
- Daily/session/Loop 延迟到对应页面。
- findings 先返回缓存和 `staleAt`，后台刷新真实 git diff。
- 增加 endpoint 级性能预算与慢查询日志。

### 6.7 P2：可访问性与状态语义

- 筛选按钮需要 `aria-pressed` 或 tab 语义。
- 图表必须有可读数据表或无障碍摘要。
- 仅用红/绿/橄榄色表达状态不够，应同时使用文本/图标。
- 表格展开行需要 `aria-expanded` 和键盘操作。
- 英文大写等宽标签较多，视觉风格明确，但降低中文用户扫读速度。

## 7. 建议的目标信息架构

## 8. 可执行改进需求

### P0-01：统一隐私数据模型

**用户故事：** 作为安全敏感用户，我要准确知道每类数据存在哪里、保留多久、是否会联网，以便信任审计结果。

**验收标准：**

- 形成机器可读的数据分类定义，覆盖 task、diff、path、secret evidence、prompt、decision reason。
- 所有 UI 隐私文案由后端能力状态生成。
- 越界记录、信箱、搜索明确显示字段来源和持久化状态。
- 提供本地数据导出、清理和保留期设置。
- 自动化测试断言默认联网 payload 不含 task/diff/code。

### P0-02：真实数据与演示数据硬隔离

**验收标准：**

- 没有真实 finding 时展示独立空状态和“加载演示”按钮。
- 演示模式使用独立路由/数据源/背景色，并在所有页面固定显示 DEMO。
- 演示项目、仓库和 finding 不参与 KPI、badge、成本或危险地图统计。

### P0-03：校准成本统计

**验收标准：**

- 为 token 累计口径补测试，防止 transcript 快照重复累加。
- 分开 input、output、cache read、cache create，并显示各自单价和计算式。
- 对异常值设置诊断提示，不静默显示。
- 能与至少一个供应商账单样本在允许误差内对账。
- 在完成校准前显示“实验性估算，不用于预算决策”。

### P0-04：处置任务状态机与 daemon 健康

**验收标准：**

- 合并终端信箱和执行记录。
- 状态至少包括 pending、claimed、running、succeeded、failed、blocked、expired、cancelled。
- 全局显示 daemon 最后心跳和最老 pending 时长。
- 支持取消、重试、改派和查看执行输出。
- 从驳回 finding 到终端执行再到守门复扫必须有端到端测试。

### P0-05：晨报保守建议规则

**验收标准：**

- 未配置预算与已用 0% 分开显示。
- 预算耗尽、数据过旧、无稳定 SHA、证据不足时不得建议 `continue`。
- 回滚点必须是完整或短 commit SHA，不得使用 `HEAD`。
- 建议必须列出依据、反证和置信度。

### P1-01：重构一级导航与渐进披露

**验收标准：**

- 默认一级导航不超过 5 项。
- Loop、成本、项目不与待处理风险平级。
- 未启用能力不显示空导航项。
- 首页 5 秒测试中，新用户能回答产品是什么、现在是否需要处理、下一步点哪里。

### P1-02：核心 finding 证据补全

**验收标准：**

- 展示稳定 commit SHA、branch、repo、agent/session、采集时间和数据新鲜度。
- 支持重新验证 finding 是否仍存在。
- 每个结论区分确定性规则、启发式判断和 AI 解释。
- 任务偏离为未知时显示 unknown，不得伪装成 0%。

### P1-03：移动端紧急审批模式

**验收标准：**

- 390px 宽度下无固定侧栏挤压和页面级横向溢出。
- finding 可完整阅读，核心按钮不被遮挡。
- 支持单手完成查看、填写理由、确认或驳回。
- 在 390×844、768×1024、1280×720 三个视口做自动截图回归。

### P1-04：高风险权限模式保护

**验收标准：**

- `bypassPermissions` 显示明确风险说明和影响范围。
- 首次启用要求二次确认，并记录 actor、时间和理由。
- 默认推荐最小权限；权限可以按项目覆盖和过期。
- 项目卡展示 daemon 是否实际采用该模式。

### P1-05：首屏性能预算

**验收标准：**

- 缓存命中时 overview/findings P95 < 500ms。
- 无缓存时先返回 stale data/skeleton，不阻塞导航。
- 非当前页面数据不参与首屏加载。
- 页面显示最后刷新时间和 stale 状态。

### P2-01：规则效果页语义修正

**验收标准：**

- 未提供编辑能力前改名为“规则效果”。
- 误报率同时显示分母和未标注状态。
- 命中占比提供文本值，不只依赖色条。
- 支持按仓库查看规则表现。

### P2-02：结构化误报反馈

**验收标准：**

- 标记误报时必须选择原因类别，可选填写说明。
- 可选择“只标注本次”或创建带作用域和期限的例外。
- 规则页能看到误报趋势和主要原因。
- AI 只能提出调级建议，不能自动修改规则。

### P2-03：搜索与快捷入口一致

**验收标准：**

- 搜索快捷跳转和实际可见导航保持一致。
- 搜索结果明确来源、时间和真实/演示状态。
- 默认不在结果摘要中展示完整任务文本。
- 支持仓库、规则、文件、commit、agent/session 过滤。

### P2-04：AI 供应商与能力动态披露

**验收标准：**

- UI 从后端读取实际 provider、base host、model、deep-code 状态。
- “已连接”基于最近一次健康检查，而非仅检查 key。
- 发送前可预览字段清单，发送后记录脱敏审计事件。
- AI 失败不影响确定性守门和人工裁决。

## 9. 建议的产品指标

不要以扫描数、wake 数或 token 数作为北极星指标。建议使用：

### 核心指标

- 每 100 次扫描触发的人工打断数。
- wake 的确认有效率：最终被驳回、修复或形成例外的占比。
- 误报率与“用户无脑放行率”。
- 从打断到完成裁决的中位时间。
- finding 发生到合并前被处理的比例。
- 同一根因重复打断次数。

### 信任指标

- 数据源新鲜度和采集器在线率。
- 可复现 finding 比例。
- 处置任务执行成功率与积压年龄。
- AI 提案被采用率，以及采用后被撤销率。
- 用户关闭 hook/规则的原因。

### 反指标

- 一级导航数量。
- 每次 wake 展示的 finding 数。
- 未经结构化反馈的简单放行比例。
- 演示数据误计入真实 KPI 的次数。

## 10. 推荐实施顺序

### 第一阶段：恢复信任

完成 P0-01 至 P0-05。目标不是加功能，而是让每个数字、每条隐私承诺和每个闭环状态都可信。

### 第二阶段：聚焦核心体验

重构导航、首页和 finding 详情；完成移动端紧急审批与首屏性能优化。

### 第三阶段：团队化

补 actor、RBAC、策略作用域、稳定 commit 证据、审批理由、导出与团队通知。

### 第四阶段：高级自治 Loop

仅在核心守门指标健康后扩展 Loop Contract、晨报和跨 agent 监控。高级能力应复用同一证据链，不再增加另一套状态系统。

## 11. 最终产品判断

这个产品真正值得做的不是“把所有 agent 活动做成一个漂亮控制台”，而是成为一个极可信、极克制的决策入口：平时消失，有问题时只拿出最重要的一两处，并能证明为什么、由谁决定、最后是否真的修好。

因此最重要的取舍是：

- 做深审查、证据、策略和闭环。
- 收缩成本、项目、Loop 和通用聊天的导航权重。
- 先解决可信度与隐私一致性，再继续扩充功能。
- 把“少打断但每次都值得”设为产品和工程共同的验收标准。
