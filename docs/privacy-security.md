# 隐私与安全 · Privacy & Security

Relay 处理高度敏感的个人数据(简历、面试记录、求职偏好)。隐私和安全是产品的一等公民,也是与竞品的差异化卖点。

## 隐私模型:数据留在设备上

核心立场:**敏感数据可以全程留在用户本地。**

- 用户 profile / 简历存 `chrome.storage.local`(浏览器扩展)
- 只有在需要 LLM 处理时,才上传**必要的最小内容**(JD + 待映射字段 + 简历相关片段)
- **密码绝不存储、绝不上传**
- 投递在用户本地浏览器完成,Relay 后端不接触用户的平台登录态

这符合 GDPR / PIPL 的数据最小化原则,"数据留在设备上"本身是营销卖点。

## 数据分层

| 数据 | 敏感度 | 存储策略 |
|------|--------|---------|
| 平台密码 | 极高 | **永不存储** |
| 简历 / 面试历史 | 高 | 可选纯本地,或加密云存 |
| 求职偏好 | 中 | 本地 + 可选云同步 |
| JD text | 低 | 不长期保存,只存提取的 metadata |
| 聚合趋势数据 | 低(已匿名) | 云存 |

## 安全机制

### 客户端
- 扩展遵循 Manifest V3 最小权限原则,只申请必要的 host permissions
- content script 与 background 之间用消息传递,不暴露敏感数据到页面上下文

### 服务器端(若用云端 LLM)
- 传输加密(TLS),静态加密(at-rest)
- LLM 请求经 proxy,API key 不下发到客户端
- 每次 agent 调用 audit log(谁、做了什么、何时)
- 行级安全(RLS):用户只能访问自己的数据

### Sandbox(服务器端工具执行)
- 每 session 独立 Docker container,网络/文件系统隔离
- 最低权限运行 + seccomp 过滤 + 资源限制
- session 结束即销毁

## 合规

- **GDPR**(欧盟):数据最小化、被遗忘权(删除)、明确同意、数据可携
- **PIPL**(中国):跨境数据处理需特别注意;中国用户数据建议本地化
- **CA AB 853** 等 AI 招聘透明度法规:持续关注,确保 AI 生成内容可被识别

## 风险边界(明确不做的事)

### ToS 风险
- LinkedIn 用户协议禁止机器人/自动访问;执法基于行为检测
- 服务器端代操作用户账户合同上有风险(参考 hiQ v. LinkedIn 的结局)
- **Relay 客户端方案规避了这个风险**——投递在用户真实环境、用户亲自提交

### 明确不实现
- ❌ 服务器端代操作用户账户(陌生 IP + 存密码)
- ❌ LinkedIn / Boss直聘 登录态自动投递(除非明确风险告知 + 用户 opt-in)
- ❌ 绕过 CAPTCHA / 反爬
- ❌ 大规模无审核轰炸投递

### 若未来实现 Phase 3 桌面 app 的批量功能
必须满足全部条件:
- 用户显式 opt-in 并同意 ToS 风险 waiver
- 投递速度强制人性化(随机延迟)
- submit 必须用户确认
- 跟踪账号封禁率,超阈值自动禁用
- 提供 dry-run 模式

## 用户控制

- 一键导出所有数据
- 一键删除账户 + 所有数据(GDPR/PIPL 合规)
- 清晰的同意流程:哪些数据上传、用于什么
- 面试数据贡献到聚合题库默认 opt-out,用户主动 opt-in

## 安全声明

本文档不构成法律建议。平台条款与 AI 招聘相关法规仍在快速演进,实现者应自行评估合规性并咨询法律专业人士。
